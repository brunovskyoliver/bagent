# WhatsApp Connector — Spike & Decision

Phase 11 integration for bagent local-first macOS assistant.

---

## Approach Comparison

| Criterion | `whatsapp-web.js` (Web bridge, v1) | Meta Cloud API (future) |
|---|---|---|
| **Auth** | QR-pair + `LocalAuth` session on disk | Business app token + webhook URL |
| **Account type** | Personal WhatsApp | WhatsApp Business only |
| **Setup cost** | Scan QR once; session auto-restores | Business verification, pricing categories, webhook infra |
| **Pricing** | Free (unofficial) | Per-conversation pricing after free tier |
| **Read chats** | Yes — all personal chats | Only messages in your Business number threads |
| **List contacts** | Yes | Only contacts that initiated conversation |
| **Fetch history** | Yes — up to WhatsApp Web limits | Limited; no retroactive history |
| **Send text** | Yes | Yes (template or session messages) |
| **Media** | Yes (deferred for v1) | Yes |
| **Official** | No | Yes |
| **Ban risk** | Present — WhatsApp ToS prohibits unofficial clients | None |
| **Session expiry** | Periodic re-link required (~14 days idle) | Token-based, long-lived |
| **DOM/API break risk** | High — WhatsApp Web internals can change without notice | Low |
| **Node/Puppeteer dep** | Required | Not needed |
| **Local privacy** | Full — messages never leave device | Messages pass through Meta's servers |

## Auth Models

### Web Bridge (`whatsapp-web.js`)
1. Daemon starts bridge subprocess; passes `BAGENT_WA_SESSION` (session dir) and `BAGENT_WA_TOKEN` (bearer).
2. Bridge initialises `whatsapp-web.js` with `LocalAuth` strategy.
3. On first run (or expired session): bridge generates a QR string, stores in memory, status = `qr`.
4. User opens Settings → WhatsApp → QR code appears → scans with phone → status = `authenticated` → `ready`.
5. Session persists at `~/Library/Application Support/bagent/whatsapp/session`.
6. On subsequent app starts: session restores without QR (if < ~14 days idle or phone re-linked).

### Cloud API (future)
1. User registers a WhatsApp Business account and creates a Meta app.
2. Provides phone number ID + permanent system user token in Settings.
3. Daemon uses the token for REST calls to `graph.facebook.com/v18.0/`.
4. Incoming messages arrive via webhook (requires public HTTPS endpoint — out of scope for local-first).

## Risks

### `whatsapp-web.js` (v1)

| Risk | Severity | Mitigation |
|---|---|---|
| **Account ban** | High | Users warned in Settings UI; no bulk/automated sends; user-approved messages only |
| **Unofficial / ToS violation** | High | Documented in Settings; user opt-in; no default activation |
| **Session expiry** | Medium | Settings shows "QR required" status; one-click refresh |
| **WhatsApp Web DOM breakage** | Medium | Version-pin; bridge degrades gracefully; connector reports `disconnected` |
| **Node/Puppeteer dep** | Low | Runtime degrade — `missing_node`/`bridge_not_installed`; install guide in Settings |
| **Chromium (~170 MB)** | Low | Explicit `make whatsapp-bridge-install` step; never auto-downloaded |
| **Local message indexing privacy** | Medium | Messages stored only on device; no cloud sync; `pii: true` in prompts; no raw bodies in audit logs |

---

## Decision

**Phase 11 v1 = local Web bridge** via `whatsapp-web.js`.

Rationale:
- Personal inbox search is the primary use case (read chats, find messages, reply).
- `whatsapp-web.js` covers this fully; Cloud API does not (Business-only, no personal history).
- Local-first architecture matches the rest of bagent's data model.
- Meta Cloud API is appropriate for a future **business connector** (e.g. "send to customers via Business number"), exposed behind a separate `WhatsAppCloudConnector` config path.

---

## Security & Privacy Contract

These constraints are enforced at the **code level** (not only policy):

1. Bridge binds **`127.0.0.1` only** — never `0.0.0.0`.
2. Bearer token (`BAGENT_WA_TOKEN`) is generated by the daemon per session; never logged, never exposed in UI.
3. **No raw WhatsApp message bodies in audit logs.** Approval audit entries use a truncated preview + char count.
4. **Every send requires explicit user approval.** The send floor is enforced in the route handler regardless of `rules.yaml` state (existing `rules.yaml` on disk may not contain the new rule).
5. No bulk sends. No group admin actions. No message deletion. No contact block/unblock. No profile/status changes. No media send in v1. No automated replies.
6. WhatsApp-derived prompt context is marked `pii: true`; the assistant summarises rather than quoting raw chats.
7. Session directory lives at `~/Library/Application Support/bagent/whatsapp/session` — same app support scope as the rest of bagent.

## Manual QA Checklist

1. Open app → Settings → WhatsApp section visible.
2. Click "Pripojiť WhatsApp" → bridge starts → status changes to "QR required".
3. QR code appears in Settings panel.
4. Open WhatsApp on phone → Linked devices → Link a device → scan QR.
5. Status changes to "Pripojené" (connected).
6. Recent chats list loads without error (check via `/whatsapp/chats` in daemon).
7. Ask: `"čo mi Peter písal o faktúre?"` → assistant searches WhatsApp cache and summarises; does **not** invent content.
8. Ask: `"napíš Petrovi na WhatsAppe: Dobrý deň, pošlem faktúru dnes večer."` → approval modal appears showing exact recipient ("Peter") and exact message text.
9. Click **Zamietnuť** (Reject) → nothing is sent; assistant confirms message was not sent.
10. Repeat step 8, click **Schváliť** (Approve) → exactly **one** message is sent; assistant confirms with delivery status.
11. Click "Odpojiť" → bridge stops → assistant responds "WhatsApp je odpojený / QR potrebné" for WhatsApp queries.
12. Restart app → session restores automatically without QR (if session still valid on phone).
13. Click "Odhlásiť a vymazať reláciu" → session cleared → QR required again on next connect.
14. Remove Node or `node_modules` → Settings shows "Node nenájdený" / "Bridge nie je nainštalovaný" status.
